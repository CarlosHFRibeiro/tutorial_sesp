SROP V5

NullPointerException could be thrown

Chamado BairroService

No método br.gov.mt.seguranca.srop.api.service.BairroService#findBairroByCoordenada, é feita uma extração do corpo do objeto HttpResponse, corpo que pode vir a ser nulo.

No mesmo método, é feita uma validação em uma string. Entretanto, diferente do caso acima, ela não pode vir a ser nula, mas pode estar vazia.

No método br.gov.mt.seguranca.srop.api.service.BairroService#geradorToken, é feita uma validação em uma string. Ela não pode vir a ser nula, mas pode estar vazia.

Solução de contorno: Não se aplica.

Solução definitiva:

  • Criar uma validação para o objeto HttpResponse.

  • Retirar da validação da string a verificação se é nula, deixando apenas a verificação se está vazia.

SonarQube

Chamado 78637

Call "Optional#isPresent()" or "!Optional#isEmpty()" Strings and Boxed types should be compared using "equals()

Existe no método br.gov.mt.seguranca.srop.api.service.BoletimOcorrenciaService#carregarTipoPessoaBoletimOcorrenciaVitima um objeto Optional, esse objeto é um encapsulador, para melhor tratamento de objetos que possam ser nulos. Uma recomendação é validação esse objeto antes de utilizar-lo, essa validação pode ser feita por if, ou métodos do próprio optional como .orElseThrow.

A segunda ocorrencia é a existencia de validação de Objetos Long utilizando ==, quando é utilizado == para comparar objetos, está verificando se as referências dos objetos são as mesmas, ou seja, se apontam para o mesmo local na memória. No entanto, quando você compara o conteúdo de objetos "boxed", o que geralmente importa é se o conteúdo é o mesmo, não se eles referenciam o mesmo objeto na memória.

Solução de contorno: Não se aplica.

Solução definitiva:

  • Utilizar alguma validação pra confirmar a existência de valores dentro do atributo Optional<Banco> banco do método br.gov.mt.seguranca.srop.api.service.BoletimOcorrenciaService#carregarTipoPessoaBoletimOcorrenciaVitima

  • Modificar a validação de == para equals

Chamado 77027

O incidente não aponta um ponto exato no código onde ocorre, não havendo um "Culprit" identificado. Além disso, não foram encontradas diferenças significativas em relação a outros campos na tela que acessem os atributos da classe. É importante notar que este incidente não é consistente, ocorrendo 5 vezes em 240118 acessos à tela no período de um mês.

Ela ocorreu quando tecnologia JSF não conseguiu iniciar a classe Action, o que por sua vez causa incidentes isolados e difíceis de reproduzir. Isso ocorre porque o JSF é baseado em componentes controladas pelo framework impedindo a interferência do desenvolvedor.

Com base nessas informações, não foi possível replicar o incidente para identificar uma solução específica.

Solução de contorno: Não se aplica.

Solução definitiva: Não se aplica.
